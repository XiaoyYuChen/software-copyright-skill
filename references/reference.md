# 软件著作权代理技能参考资料（详细版）

本文件提供详细规则、枚举清单与审核细则。  
主流程以 `SKILL.md` 为准；仅在以下场景读取本文件：

- 需要完整字段清单与字数限制明细
- 需要完整技术特点或编程语言候选列表
- 需要详细审核条目逐项对照
- 需要补正建议的标准化措辞

---

## 功能概述（详细）

本技能具备两大核心功能：
1. **制作申请材料**：帮助用户制作完整的软件著作权申请材料，包括基本信息采集、程序鉴别材料和软件鉴别材料的整理。
2. **审核申请材料**：对他人提交的软件著作权申请材料进行全面审核，检查材料完整性、规范性和合规性。

### 默认配置

- **软件开发配置**：默认使用当前开发环境信息
- **执行配置**：默认使用当前开发环境作为运行环境

技能会自动检测当前项目的开发环境，并将其作为软件著作权申请的默认配置，用户可根据实际情况进行修改。

## 申请材料结构（详细）

### 1. 基本信息

#### 软件全称规范

- 软件全称应使用规范命名格式：**XXX软件**、**XXX系统**或**XXX平台**
- 文档、源代码页眉的软件名称需与申请表保持一致
- 所有材料中的软件名称必须统一，不得出现差异

#### 开发环境信息

| 项目 | 说明 | 字数限制 |
|------|------|----------|
| 开发的硬件环境 | 开发软件所使用的硬件设备（必须含 CPU/内存/硬盘/显卡；不含显示器） | 50字 |
| 运行的硬件环境 | 软件运行所需的硬件设备（必须含 CPU/内存/硬盘/显卡；不含显示器） | 50字 |
| 开发该软件的操作系统 | 开发过程中使用的操作系统 | 50字 |
| 软件开发环境 / 开发工具 | 使用的开发工具和环境 | 50字 |
| 该软件的运行平台 / 操作系统 | 软件运行的目标操作系统 | 50字 |
| 软件运行支撑环境 / 支持软件 | 运行所需的支持软件 | 50字 |

#### 输入限制检查

技能会自动检查所有输入内容是否符合网页输入限制：

1. **实时字数统计**：在用户输入时实时显示已输入字数和剩余字数
2. **自动截断**：超出限制时自动截断并提示用户
3. **智能压缩**：对于接近限制的内容，提供精简建议
4. **格式验证**：检查特殊字符、格式是否符合要求
5. **预填充验证**：自动填充的内容也会经过严格验证

#### 字数限制明细

| 字段类别 | 字数限制 | 说明 |
|----------|----------|------|
| 开发环境信息（每项） | 50字 | 如开发硬件环境、运行硬件环境等；硬件项必须含 CPU/内存/硬盘/显卡，不含显示器 |
| 自定义编程语言 | 120字 | 其他编程语言输入限制 |
| 开发目的 | 50字 | 开发软件的原因和目标 |
| 面向领域/行业 | 50字 | 软件应用的行业领域 |
| 软件主要功能 | 500-1300字 | 核心功能描述 |
| 技术特点描述 | 100字 | 自定义技术特点说明 |

#### 硬件环境填写规范（新增）

- 开发硬件环境和运行硬件环境都必须至少包含以下 4 项：
  - CPU
  - 内存
  - 硬盘
  - 显卡
- **不需要**填写显示器、屏幕尺寸、分辨率等显示外设信息；若用户填写了显示器相关内容，应删除并提示。
- 若用户未提供完整四项，必须继续追问，不可跳过。
- 若字数超限（50字），优先保留四项核心参数并压缩修饰词。

### 2. 程序鉴别材料

#### 交存方式
- 一般交存
- 例外交存

#### 提交要求
- 提交登记软件源程序连续的前30页和连续的后30页（前、后各连续30页可以按开发时间排序，也可以按功能主次等自定义排序）
- 每页不少于50行
- 若源程序整体不到60页，应提交全部源程序
- 格式：PDF
- 文件大小：建议不超过50MB

#### 源码采集范围与优先级（新增）

生成程序鉴别材料时，`scripts/generate_program_pdf.py` 按以下原则选取源程序（与 `SKILL.md` 一致）：

| 优先级 | 类型 | 说明 |
|--------|------|------|
| 高 | 用户业务源码 | `src/`、`app/`、`pages/`、`components/`、`api/`、`services/` 等目录中的源文件（多语言扩展名，**不含** `.json` 配置） |
| 中 | 其它自研源码 | 未落在排除规则中的项目内源文件 |
| 低（补足用） | 测试 / 示例 / 文档目录 | `test/`、`__tests__/`、`spec/`、`examples/`、`docs/` 等，仅当高优先级源码凑不满 60 页×50 行时才会排到后面 |

**一律排除**（不得出现在程序鉴别材料中）：

- 配置文件：`package.json`、`tsconfig.json`、`manifest.json`、`pages.json`、`*.config.js`、`vite.config.*`、`.env*` 等
- 数据与锁文件：`*.lock`、`requirements.txt`、`composer.lock`、数据集 / `fixtures/` / `migrations/` / `seeds/` 等目录
- IDE 与工具目录：`.idea/`、`.vscode/`、`.hbuilderx/`、`.vs/` 等
- 构建与依赖产物：`node_modules/`、`dist/`、`build/`、`unpackage/`、`target/`、`__pycache__/`、`generated/`、`__generated__/` 等
- 压缩与生成物：`*.min.js`、`*.bundle.js`、`*.map`、`*.d.ts`（声明/自动生成）、`*.g.dart` 等

#### 代码排版与标识规则（新增）

- 固定总页数：60 页（连续前 30 + 连续后 30）；第 1–59 页每页 50 行有效代码，第 60 页至少 40 行且以完整模块结尾。
- 页眉：居中「软件全称+版本号」（`--project-name` + `--version`，默认 `V1.0`）；右上角页码「第 N 页」。
- **不输出任何行号**（无源文件行号、无页内 01~50 前缀）。
- 每行格式必须为：`文件名:  代码`（冒号后固定两个空格）。
- 左侧标识规则：
  - 代码行固定仅显示文件名，不显示目录路径。
  - 目录归属信息通过模块行体现（见下文）。
  - 文件名最大长度 30 字符，超出时保留尾部并前置省略号。
- 模块行规则：
  - 每个文件首行之前插入一行 `模块:<文件夹路径>`。
  - `模块:` 行不计入每页 50 行代码统计。
  - 模块行用于人工审核目录归属，建议在导出文档中视觉强化（如加粗或字号增大）。
- 一致性规则：
  - `程序鉴别材料.md`、`code-pages-60.html`、`程序鉴别材料.pdf` 必须从同一份源数据生成，避免版本偏差。
- 名称替换规则：
  - 模板输出阶段若出现历史名称（如“艾宾浩斯背诗工具”）或真实项目名称，应自动替换为 `PROJECT_NAME`。
  - 标题、页眉、说明文本中的软件名必须与软件全称一致。
- 注释处理规则：
  - 程序鉴别材料输出前应移除注释内容（如 `//`、`#`、`/* ... */`、`<!-- ... -->`）。
  - 仅输出去注释后的有效代码行，材料中不标注行号。

#### 内置生成脚本（与 SKILL.md 一致）

位于 skill 的 `scripts/` 与 `assets/`，生成时**必须复用**，禁止在业务项目里复制改逻辑（除非用户明确要求 fork）：

| 路径 | 说明 |
|------|------|
| `scripts/generate_program_pdf.py` | 主程序：`--project-root`、`--project-name`、`--version`（固定 60 页） |
| `scripts/程序鉴别材料-转PDF.bat` | `bat <项目根> [软件全称]` → 调用上述脚本 |
| `assets/code-pages-60.html` | skill 内排版样例；正式输出路径为 `<项目>/software-copyright/code-pages-60.html` |

命令示例：

```bash
python ~/.cursor/skills/software-copyright-skill/scripts/generate_program_pdf.py \
  --project-root /path/to/repo \
  --project-name "XXX系统" \
  --version "V1.0"
```

依赖：`reportlab`；中文页眉页脚使用 `C:\Windows\Fonts\simhei.ttf`（不存在时回退 CID 字体）。

### 3. 文档鉴别材料

> **生成时必须使用 skill 模板**：[assets/说明文档模板.doc](../assets/说明文档模板.doc)、[assets/说明文档模板.md](../assets/说明文档模板.md)。完整细则见 [doc-template.md](doc-template.md)。

#### 模板要求摘要

- 类型：《说明文档》，图文结合的操作说明
- 页数：**超过 7 页**，保证内容丰富和完整，**不超过 60 页**（硬性要求）
- 页眉：软件全称+版本号（居中）；右上角页码
- **标题**：PDF 文首软件全称须**居中**显示
- 结构：第一章 系统描述（1.1 系统价值、1.2 系统功能描述）→ 第二章 系统模块介绍（登录/主界面/各功能 + 截图）
- 有登录从登录界面开始；正式件删除文首注意事项

#### 交存方式（登记中心一般交存）

- 提交登记软件文档连续的前 30 页和后 30 页，每页不少于 30 行
- 若整个文档不到 60 页，应提交整个文档
- 格式：PDF；建议不超过 50MB

#### 文档定义
文档是指用来描述程序的内容、组成、设计、功能规格、开发情况、测试结果及使用方法的文字资料和图表等，如程序设计说明书、流程图、用户手册等。

#### 软件运行截图要求

文档鉴别材料必须包含软件运行截图，须符合说明文档模板：

- 每个功能模块：文字说明 + 对应截图 + **图注**（截图下方居中，如 `图1-首页`）
- 截取真实运行界面，清晰完整
- 流程连贯；有登录时从登录开始
- 无操作界面时可提供流程图
- 覆盖申请表中的主要功能与技术特点

### 4. 其他相关证明文件

- 格式：PDF
- 文件大小：建议不超过50MB
- 可添加多个文件

## 交互机制（详细）

### 智能询问原则

技能采用智能询问机制，**仅在必要时询问用户**，而非每项都问：

1. **项目文件扫描**：在询问用户前，先自动扫描项目文件获取信息：
   - `package.json`：获取项目名称、描述、依赖、脚本等
   - `pom.xml`：获取项目名称、描述、依赖等
   - `requirements.txt`：获取Python依赖信息
   - `README.md`：获取项目描述、功能说明等
   - 项目目录结构：推断项目类型和规模
   - 源代码文件：识别编程语言和技术栈
2. **自动获取信息**：优先从当前开发环境和项目文件自动获取配置信息
3. **智能推断**：根据已获取的信息推断相关字段（如根据编程语言推断技术特点）
4. **仅问必要项**：仅在以下情况询问用户：
   - 无法从项目文件获取且无默认值的必填项
   - 需要用户确认或选择的关键选项
   - 存在歧义需要澄清的内容
5. **提供建议选项**：对于可选字段，列出推荐选项供快速选择
6. **给出参考说明**：对于专业术语或复杂选项，提供解释和示例
7. **支持自定义输入**：除预设选项外，允许用户输入自定义内容
8. **保存草稿功能**：支持中途保存，方便用户分多次完成

### 询问注意事项

询问用户时需避免以下低级错误：

1. **不要询问可自动获取的信息**：
   - **开发语言**：从源代码文件扩展名和配置文件自动识别（如 `.py` -> Python）
   - **代码量**：自动统计所有源代码文件行数
   - **开发环境**：从配置文件自动获取（如 `package.json`、`pom.xml`）
   - **运行环境**：从配置文件和 Docker 配置自动识别
2. **不要重复询问**：
   - 已确认的信息不再重复询问
   - 自动填充的信息先展示给用户确认，而非直接询问
3. **不要使用模糊表述**：
   - 使用明确的问题，避免歧义
   - 提供具体示例帮助用户理解
4. **不要遗漏重要信息**：
   - 确保所有必填项都被覆盖
   - 按逻辑顺序提问，避免跳步
5. **不要忽略格式限制**：
   - 提前告知用户字数限制
   - 自动检测并提示超出限制的内容

## 审核功能（详细）

> **完整审查清单**见 [review-checklist.md](review-checklist.md)（实名认证、附加证明 1–5、源码文档、操作文档）。审核模式**必须先读**该文件。

### 审核流程

1. 确认著作权人已在版权中心**实名认证**（<https://register.ccopyright.com.cn/login.html>）
2. 根据申请情形核查**附加证明文件**（合作/委托/前期开发/非职务/上级法人）是否触发且齐全
3. 逐项审核基本信息、程序鉴别材料、文档鉴别材料
4. 生成审核报告和整改建议（缺失附加证明标 **[高]**）
5. 用户修改后复审

### 附加证明文件（触发则必填）

详见 [review-checklist.md](review-checklist.md) 第一节。摘要：符合情形 1–4 **缺少则无法申请**。

### 审核要点

#### 基本信息审核
- 软件全称是否符合规范（XXX软件/XXX系统/XXX平台）
- 所有材料中软件名称是否一致
- 开发环境信息是否完整准确
- 编程语言是否正确填写
- 软件主要功能描述是否符合要求（500-1300字）
- 技术特点是否与实际功能匹配

#### 程序鉴别材料审核
- 是否提交前30页和后30页源程序：若未提交完整，需补充缺失部分
- 每页是否不少于50行：不足50行需补充内容或重新排版
- 源程序不足60页时是否提交全部：需确认并提交完整源程序
- PDF格式是否正确，文件是否清晰可读：格式错误需重新生成PDF
- 源代码页眉软件名称是否与申请表一致：不一致需修改页眉，确保与申请表软件名称完全相同
- 页眉是否为「软件全称+版本号」、右上角是否有页码（第 N 页）

#### 文档鉴别材料审核
- 《说明文档》页数是否 **>7 且 ≤60**：≤7 页须扩充内容与截图；>60 页须精简
- 是否按说明文档模板两章结构编写、是否已删除文首注意事项
- **PDF 标题（软件全称）是否居中**
- 每页是否不少于30行：不足30行需补充内容或重新排版
- 文档不足60页时是否提交全部：需确认并提交完整文档
- 是否包含软件运行截图：**缺少截图需补充软件真实运行的完整、清晰界面**，展现连贯使用流程和全部功能
- 截图是否展现完整功能和使用流程：截图不完整需重新截取；**每张图下方是否标注图注**（`图N-简述`）
- PDF格式是否正确，文件是否清晰可读：格式错误需重新生成PDF
- 文档页眉软件名称是否与申请表一致：不一致需修改页眉，确保与申请表软件名称完全相同
- 页眉是否为「软件全称+版本号」、右上角是否有页码（第 N 页）

#### 其他材料审核

按 [review-checklist.md](review-checklist.md) 核查：合作开发协议、委托开发协议、前期开发说明、非职务开发声明、上级法人单位证明；PDF 格式与大小。

### 审核报告

审核完成后生成详细报告，包含：
- 审核结果（通过/需修改/不通过）
- 问题清单及整改建议
- 优先级标注（高/中/低）
- 修改示例和参考标准
- 复查时间建议

### 审核交互机制

- 发现问题时主动标注并说明原因
- 提供修改建议和参考示例
- 对不确定的内容询问用户确认
- 支持多次审核和修改跟踪

## 输出目录规划（详细）

目录结构：

```text
项目根目录/
├── software-copyright/
│   ├── 基本信息.md
│   ├── 程序鉴别材料.md
│   ├── 文档鉴别材料.md
│   ├── 补正.md
│   └── README.md
```

目录位置选择：
1. 优先检测 `Doc/`、`Docs/`、`项目文档/`
2. 若不存在，则在项目根目录创建 `software-copyright/`

README.md 需包含（当无法直接生成合规PDF时）：
- 页眉设置步骤
